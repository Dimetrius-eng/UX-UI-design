**2.1. Чим Task Flow відрізняється від User Flow?**

**User Flow** — це загальний шлях користувача до досягнення певної цілі в продукті. Він показує широку картину взаємодії з системою.  
**Task Flow** — це послідовність дій у межах одного конкретного завдання. Він концентрується на дрібніших задачах (наприклад, пошук за локацією або редагування кошика).  
Основна різниця між ними полягає в масштабі. Розбиття на Task Flow дозволяє працювати з маленькими фрагментами інформації, щоб загальна схема не набула «монструозних розмірів» і залишалася зрозумілою.

**2.2.** ***Чому вайрфрейми зазвичай виконуються у відтінках сірого?**

По перше, вайрфрейми фокусуються на структурі. Обмеження чорно-білою або сірою палітрою допомагає зосередитися на розташуванні елементів та формуванні інформаційної ієрархії, а не на візуальному оформленні. По друге, вони зменшують когнітивне навантаження через відсутність кольорів, що дозволяє команді та замовнику не відволікатися на естетику і приймати рішення щодо функціональності та логіки інтерфейсу. І нарешті, у такий спосіб вони економлять час, що дозволяє швидше вносити правки та експериментувати з концепціями на ранніх етапах.

**2.3.** ****Що таке «Grey Box Method» у прототипуванні?**

**Grey Box Method** — це техніка створення низькодеталізованих вайрфреймів (Low-fi), де всі елементи інтерфейсу та контент зображуються у вигляді простих сірих фігур.

**4. Дослідіть питання, які є рекомендації щодо створення  вайрфреймів з Layout Grid для різних пристроїв? Як ви їх врахували у своєму дизайні?**

При проєктуванні вайрфреймів для основного сценарію я дотримувався певних приципів. Так, оскільки цільовою платформою є мобільний пристрій, я застосував 4-колонкову сітку (Column grid). Це дозволило уникнути хаотичності та забезпечило «повітря» в інтерфейсі. Усі інтерактивні елементи, включаючи кнопки, суворо вирівняні по межах колонок сітки. Використання Layout Grid дозволило створити логічний та масштабований «скелет» продукту, який легко адаптувати під різні екрани та передати в подальшу розробку.

**2. Вкажіть розміри основних елементів, які були використані у вашому дизайні.**

Для проєкту фітнес-застосунку було обрано компактну типографічну шкалу, що базується на модульному коефіцієнті 1.25 (Major Third) для переходу від базового тексту до підзаголовків. Хоча розміри менші за стандартні вебрекомендації, це дозволяє розмістити більше функціональних елементів на одному екрані мобільного пристрою без необхідності зайвого скролінгу.   

Технічні параметри (кегль та інтерліньяж):
- **Кегль (font size)**. Основний текст зафіксовано на рівні 12 px, що є мінімально допустимим порогом читабельності для мобільних інтерфейсів.  
- **Інтерліньяж (line height)**. Для покращення сприйняття тексту в умовах обмеженого простору використано інтерліньяж 1.4 (17 px) для основного тексту та 1.2 (25 px) для заголовків H1 .
Це дозволяє рядкам не «злипатися» та полегшує сканування інформації оком.

Візуальна диференціація:  
Оскільки заголовки H2 та H3 мають однаковий кегль (15 px), їх розділено візуально за допомогою накреслення (Font Weight) та регістру.
- **H2 (Congratulations!):** використано напівжирне накреслення (Bold) та центральне вирівнювання для екрана Success State.  
- **H3 (Push-ups)**: використано середнє накреслення (Medium) у комбінації з іконками вправ для створення чіткої ієрархії всередині карток.

### Контрольні запитання

**1. Який елемент вайрфрейму вказує на те, що тут буде зображення?**

Для позначення місця під зображення (фотографію, ілюстрацію або банер) у вайрфреймах прийнято використовувати прямокутник, перекреслений двома діагональними лініями. 
Такий плейсхолдер чітко вказує на наявність графічного контенту, не відволікаючи увагу на його деталі.

**2. Навіщо використовувати реальні тексти замість "Lorem Ipsum" на етапі High-fi вайрфреймів?**

Використання реальних текстів на етапі High-fidelity дозволяє перевірити ієрархію, заголовки та основний текст мають бути жирнішими та більшими, щоб користувач міг легко зчитати структуру. Також реальний контент допомагає зрозуміти, чи вміщується інформація в блоки та чи не порушується композиція при зміні довжини слів. Користувач або замовник краще розуміють цінність функцій, коли текст відповідає реальним завданням.

**3. Що таке "Happy Path" у сценарії користувача?**

**Happy Path (щасливий шлях)** — це ідеальний сценарій взаємодії, за якого користувач досягає своєї цілі найкоротшим шляхом без жодних помилок, виняткових ситуацій або відхилень. Наприклад, у моєму застосунку це успішна реєстрація з першої спроби та миттєвий перехід до тренування.

**4.** ***Як Layout Grid допомагає у передачі макета розробнику?**

Сітка (Layout Grid) виступає математичною основою макета, яка стандартизує відступи (розробник бачить чітку систему колонок та міжрядкових інтервалів, що спрощує верстку), забезпечує адаптивність (сітка допомагає передбачити, 
як блоки будуть перебудовуватися при зміні розміру екрана (наприклад, з мобільного на десктоп)), виключає хаос (елементи, вирівняні по сітці, виглядають цілісно, що робить передачу макета в розробку зрозумілішою).

**5.** ****Чому важливо проєктувати екран "Success State" (успіх), якщо дія відбувається на сервері (наприклад, надсилання листа)?**

Навіть якщо технічна частина завершується на сервері, користувач має отримати візуальний зворотний зв'язок. Проєктування «Success State» дозволяє дати відчуття завершеності дії (наприклад, тренування закінчено, дані збережено), запобігти повторним діям користувача через непевність (наприклад, повторне натискання кнопки «Відправити») та покращити емоційний досвід, підкреслюючи успіх користувача.
